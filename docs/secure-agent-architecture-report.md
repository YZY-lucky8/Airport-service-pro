## 一、核心结论：为什么构建安全 Agent 如此困难？

根据 **2026年6月** 最新研究（南洋理工大学 & IBM Research），即便是最顶级的模型（如 GPT-5、Gemini），在面对**提示词注入攻击（Prompt Injection）**时，直接攻击成功率仍高达 **79%**，隐藏攻击（嵌入网页内容）成功率也达到 **68%**。

这意味着：**传统的“输入过滤”已经彻底失效**。攻击者不再直接对抗模型，而是通过污染 Agent 检索到的外部信息（如网页、数据库），让 Agent "自愿"执行恶意指令。

### 🚀 应对方案
要构建一个安全的智能体，不能只靠模型本身，必须构建一个**纵深防御体系（Defense in Depth）**，包含：
1. **感知层防御**：多 Agent 协同检测注入攻击。
2. **执行层验证**：形式化验证（VeriGuard）确保 Agent 行为不偏离安全边界。
3. **通信层隔离**：防止 Agent 之间的“提示词感染”（Prompt Infection）。

## 二、顶刊论文核心洞察（前沿防御技术）

### 1. Google VeriGuard：形式化验证安全边界
**论文：** *VeriGuard: Enhancing LLM Agent Safety via Verified Code Generation (2025)*  
**核心原理：**  
传统的 Agent 在运行时才判断是否违规，而 VeriGuard 提出了**“离线合成 + 在线监控”**的双阶段架构：
- **离线阶段**：根据用户需求合成一套"行为策略代码"，并进行**形式化验证**，确保代码在数学上绝对符合安全规范。
- **在线阶段**：Agent 的每一个动作在执行前，都会被这个经过验证的代码拦截检查。
**启示：** 在关键业务（如机场自助机的航班修改、行李托运），不应让 Agent 直接操作数据库，而应通过预验证的代码接口执行。

### 2. 多 Agent 防御流水线：让 AI 保护 AI
**论文：** *A Multi-Agent LLM Defense Pipeline Against Prompt Injection Attacks (IEEE WIECON-ECE 2025)*  
**核心原理：**  
使用一组**专门的防御 Agent** 来拦截攻击。实验证明，这种流水线架构能将攻击成功率从 20-30% 降至 **0%**。
- **结构：** 原始 Agent 负责干活；防御 Agent 负责"看住"原始 Agent。

### 3. ICLR 2025 Prompt Infection：Agent 间的“病毒传播”
**论文：** *Prompt Infection: LLM-to-LLM Prompt Injection within Multi-Agent Systems*  
**核心风险：**  
在多 Agent 协作系统中，一个 Agent 可能被注入恶意指令，然后把这个指令**伪装成有用信息**传递给另一个 Agent，导致恶意指令像病毒一样扩散。
**防御方案：** 提出 **LLM Tagging** 机制，在 Agent 间传递信息时，为每一段内容打上“信任标签”，明确区分“系统指令”和“用户数据”。

## 三、框架选型：为什么 LangGraph 是安全 Agent 的首选？

在 2026 年，主流 Agent 框架主要有三个：**LangGraph**、**CrewAI**、**AutoGen (AG2)**。

| 维度 | LangGraph | CrewAI | AutoGen |
|------|-----------|--------|---------|
| **核心架构** | **有向图状态机 (DAG)** | 角色扮演团队 (Role-play) | 会话驱动的协作 (Chat-based) |
| **可控性** | **极高**（每个节点和边都由代码精确控制） | 中（模型自主权大，容易发散） | 低（Agent 之间聊天太自由，难以审计） |
| **状态管理** | **显式状态快照 (Checkpointing)** | 隐式 | 隐式 |
| **安全审计** | **优秀**（每一步都可中断、暂停、人工介入） | 较差 | 较差 |
| **适用场景** | **高合规、高安全、复杂分支逻辑** | 内容创作、快速原型 | 研究性任务、代码协作 |

### 🔒 安全结论：选择 LangGraph
构建安全 Agent 的核心是**“确定性”**。LangGraph 基于**有限状态机**，Agent 的每一步跳转（Edge）都是由明确的逻辑代码（而非模型幻觉）决定的。这意味着你可以把“安全检查节点”硬编码在图中，Agent 跳不过去。

### 2. 四大核心模块实现逻辑

#### A. 感知模块 (Perception) + 数据隔离
- **功能**：接收用户输入（文本/语音）。
- **安全设计**：**User Context Isolation（用户上下文隔离）**。
- **实现**：每个用户的请求必须绑定唯一的 `userId`。Agent 在查询知识库或数据库时，SQL 语句中必须强制注入 `WHERE user_id = '{userId}'`，防止横向越权。

#### B. 规划模块 (Planning) + 状态机
- **功能**：决定下一步做什么。
- **安全设计**：**Constrained Decoding（受限解码）**。不使用自然语言描述下一步，而是让模型从预定义的“动作列表”中做**单选题**。
- **示例**：模型只能输出 `{"action": "check_flight", "params": {"flight_no": "CA1234"}}`，不能输出其他随意文本。

#### C. 执行模块 (Action) + 沙箱机制
- **功能**：调用 API 或数据库。
- **安全设计**：**Function Calling with Schema（基于 Schema 的函数调用）**。
- **实现**：定义严格的 JSON Schema。如果模型传入的参数类型不对（如航班号传了数字），框架自动拒绝执行。

#### D. 记忆模块 (Memory) + 遗忘机制
- **功能**：存储上下文。
- **安全设计**：**Ephemeral Memory（临时记忆）**。
- **实现**：机场自助机场景不需要 Agent 记住用户的一生。每次会话结束后，内存自动清空（Garbage Collection），只保留审计日志。

## 五、在“机场自助机”项目中的落地方案

我们可以将这套架构无缝接入你现有的 `Airport-service-pro` 项目。

### 改造方案：

1. **后端重构 (Node.js + LangGraph.js)**
   - 目前你的 `qa_knowledge` 是简单的关键词匹配。我们将引入一个轻量级的 **Intent Classifier**（意图分类器）作为入口。
   - 引入 **LangGraph.js**（JavaScript 版本的 LangGraph）管理流程。

2. **安全拦截器中间件 (Security Middleware)**
   在 Express 的中间件层加入一个 `SecurityGate`：
   ```javascript
   // 伪代码
   app.use('/api/agent-action', async (req, res, next) => {
       // 1. 输入过滤：检测 Prompt Injection
       if (llmGuard.detect(req.body.input)) {
           return res.status(403).send("检测到恶意指令");
       }
       
       // 2. 权限验证：检查该 userId 是否有权限修改航班
       const hasPerm = await checkPermission(req.userId, req.body.action);
       if (!hasPerm) {
           return res.status(403).send("权限不足");
       }
       next();
   });
   ```

3. **日志审计 (Audit Trail)**
   利用你现有的 `audit_logs` 表，记录 Agent 的每一个思考节点：
   ```sql
   INSERT INTO agent_audit_logs (
     request_id, user_id, node_name, input_summary, output_decision, timestamp
   ) VALUES (
     'req-001', 'U123', 'check_flight', 'User asked for CA1234 status', 
     'Action: QueryFlightDB', NOW()
   );
   ```
